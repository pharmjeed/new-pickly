# 03 — رحلات المستخدمين (Customer Journeys)

المصدر السابق: 00–02 · الشاشات المشار إليها بأرقامها من `04-screen-inventory.md`

---

## J1. السيناريو الطبيعي الكامل (Happy Path) — الدفع بعد القبول (قرار المالك 2026-07-11)

1. العميل يتصفح (زائر أو مسجل) → يختار مطعماً وفرعاً → يخصص منتجات → سلة → يسجل/يدخل → يحدد سيارته → **يرسل الطلب بلا دفع**.
2. الطلب يصل للفرع (تنبيه صوتي + عداد قبول BR-1).
3. الفرع يقبل محدداً **وقت التجهيز المتوقع** (10/15/20/25 د)؛ العميل يصله إشعار «قبل المطعم — الوقت X دقيقة، يناسبك؟».
4. العميل يوافق على الوقت (مهلة 5 د) → يُحوَّل لصفحة الدفع → يدفع (مهلة 5 د). لم يناسبه الوقت؟ إلغاء بلا أي رسوم.
5. **نجاح الدفع = لحظة الصفر**: الطلب ينتقل فوراً لخانة التحضير، وعداد دقائق التجهيز يبدأ، والعميل يضغط «انطلقت الآن» → Pickly يحسب ETA من الطريق الفعلي.
6. الفرع يرى في **رادار الوصول** كل عميل متجه: المتبقي لوصوله مقابل ما مضى من دقائق التجهيز المتفق عليها، ويتلقى إشعارات الاقتراب (10/5/3 دقائق) وتنبيهاً أحمر عند **≤ دقيقة أو الوصول**.
7. العميل يصل → يؤكد «وصلت» → يحدد الموقف → يؤكد السيارة.
8. بطاقة كبيرة على شاشة الفرع → «خرج الموظف» → العميل يرى «الموظف متجه إليك».
9. التحقق (رمز 4 أرقام أو QR أو زر «استلمت») → تم التسليم → فاتورة + تقييم + بقشيش اختياري.

## J2. أول استخدام

Splash → اللغة → 3 شاشات تعريف (اطلب مسبقاً/توجه/استلم داخل سيارتك) → المدينة/الموقع (بشرح الفائدة، بلا إجبار) → **تصفح كزائر مسموح** → عند أول دفع: جوال + OTP + الاسم → إضافة السيارة (نوع، شركة، موديل اختياري، لون، آخر 4 أرقام لوحة، صورة وملاحظات اختيارية، افتراضية) → إعداد الإشعارات بشرح السبب.

## J3. الطلب المجدول

السلة → «تحديد وقت لاحق» → اليوم/التاريخ → فترات متاحة بسعة لكل فترة → آخر موعد للتعديل → **إرسال بلا دفع** (الحجز يؤكد السعة) → يظهر في «مجدولة» + خيار إضافته للتقويم → قبل الموعد: تذكير «حان وقت التوجه» → عند دخول الفترة يدخل مسار J1 من الخطوة 2 (قبول الفرع ثم موافقة العميل على الوقت ثم الدفع).

## J4. الويب (بلا تطبيق)

رابط مطعم أو رابط متابعة طلب → نفس التصفح والسلة والدفع من المتصفح → **Web Check-in**: «أنا في الطريق» و«وصلت» وتحديد الموقف من الصفحة → لا يُشترط تحميل التطبيق في أي خطوة.

## J5. رفض المطعم / انتهاء مهلة القبول

رفض بسبب مغلق (منتج غير متوفر، فرع مغلق، ضغط مرتفع، مشكلة نظام، آخر) أو صمت حتى انتهاء العداد → إلغاء آلي وفق السياسة → **لا أثر مالي — الرفض يسبق الدفع** (الاسترجاع فقط لطلب قديم دُفع قبل تغيير المسار) → إشعار العميل → تسجيل السبب واحتسابه في تقييم الفرع التشغيلي.

## J6. منتج غير متوفر بعد القبول

الفرع يحدد المنتج → خيارات للعميل (شاشة 40): قبول بديل (مع فرق السعر) / حذف المنتج واسترجاع قيمته / إلغاء الطلب → لا تعديل سعر دون موافقة العميل → تحديث الفاتورة والاسترجاع الجزئي آلياً.

## J7. العميل وصل قبل الجاهزية

يظهر للفرع «وصل مبكراً» بوضوح مع الوقت المتبقي → يظهر للعميل الوقت المتوقع → **لا يُرسل موظف قبل الجاهزية** → وقت انتظاره يُحتسب منفصلاً عن تأخير الفرع في المقاييس.

## J8. العميل تأخر بعد الجاهزية

تذكير بعد مدة محددة → المحافظة على الطلب وفق صلاحية المنتج (سياسة خاصة للأطعمة الحساسة للحرارة) → بعد العتبة: تصنيف No-show وفق السياسة (`06-business-rules.md`).

## J9. العميل ذهب للفرع الخطأ

النظام يكتشف تعارض موقع العميل مع الفرع المستهدف → رسالة واضحة باسم الفرع الصحيح + فتح الاتجاهات → **منع إرسال إشارة وصول خاطئة للفرع الآخر**.

## J10. تعطل الموقع / رفض الصلاحية

كل شيء يعمل يدوياً: «أنا في الطريق» → إدخال وقت وصول يدوي → «وصلت» → اختيار الموقف + وصف موقع السيارة + صورة اختيارية → الاتصال بالفرع. الموقع ميزة تحسين، ليس شرط تشغيل.

## J11. مشكلة في التسليم (جانب الموظف)

«لم أجد العميل» → محاولة اتصال → تصعيد لمدير الفرع → توثيق handoff_attempts. بيانات سيارة خاطئة / طلب تالف / عنصر ناقص → مسارات موثقة تفتح بلاغاً وترتبط بالاسترجاع.

## J12. الإبلاغ والاسترجاع (جانب العميل)

بعد الطلب → «الإبلاغ عن مشكلة» (منتج ناقص/خاطئ/جودة/تأخر/لم يصل الموظف/مبلغ خاطئ/مشكلة دفع/طلب لم أقم به) → اختيار العناصر المتضررة + صورة + تفاصيل → استرجاع كامل/جزئي وفق السياسة → تذكرة بمتابعة حالة → قرار (تاجر أو Pickly) → تنفيذ الاسترجاع وإشعار.

## J13. انضمام تاجر جديد

من الموقع التعريفي → إنشاء حساب + بيانات المنشأة (سجل، ضريبي، مسؤول، بنك) → مراجعة المستندات (Merchant Success) → تحقق بنكي → عقد → إضافة الفروع وإعدادات الاستلام والمواقف → ربط POS أو قائمة يدوية → **طلب تجريبي كامل** → تفعيل. الصفحة تعرض نسبة اكتمال الإعداد.

## J14. الوردية اليومية للفرع

دخول (كود فرع + حساب/PIN + ربط الجهاز) → فتح الوردية (المسؤول، حالة القبول، مستوى الازدحام، زمن التجهيز، المواقف) → تشغيل اليوم → إغلاق الوردية (الطلبات المفتوحة وغير المسلمة، ملخص الأداء، تسليم الوردية التالية).

## J15. وضع الازدحام

مدير الفرع يفعّل → زيادة وقت التحضير (+10/20/30 د) أو إيقاف مؤقت أو حد أقصى للطلبات أو إغلاق الاستلام فقط + رسالة تظهر للعملاء → ينعكس فوراً على الاكتشاف والسلة.
