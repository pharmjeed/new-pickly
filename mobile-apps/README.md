# تطبيقات بيكلي للجوال — العميل + الفرع (أندرويد وآبل)

هذان تطبيقان **قابلان للتثبيت** (لهما أيقونة على شاشة الجهاز) يعرضان تجربة بيكلي المصقولة الحيّة نفسها
بكامل مزاياها داخل غلاف أصلي رفيع (React Native / Expo + WebView). يعملان على **جوال وتابلت** أندرويد وآبل.

| التطبيق | المجلد | يفتح | المعرّف (bundle) | الأيقونة |
|---|---|---|---|---|
| العميل | `mobile-apps/customer` | `https://thepickly.com` | `sa.pickly.customer` | القرطاس المبتسم (ليموني) |
| الفرع | `mobile-apps/branch` | `https://branch.thepickly.com` | `sa.pickly.branch` | شبكة اللوحة (كحلي) |

**ما يضيفه الغلاف فوق الويب:** إذن الموقع الأصلي لتدفّق «وصلت» (العميل)، فتح خرائط قوقل/الهاتف خارج التطبيق،
زر الرجوع في أندرويد، سحب-للتحديث، شاشة «لا اتصال» بإعادة محاولة، وحفظ جلسة الدخول (OTP) بين التشغيلات.

**تطبيق الفرع تحديداً يضيف أيضاً (منذ 2026-07-16):**
- **الشاشة لا تقفل نفسها** ما دام التطبيق في الواجهة (expo-keep-awake) — التابلت شاشة تشغيل دائمة.
- **إشعار نظامي «طلب جديد وصل» يرن حتى والجهاز مقفول** (expo-notifications): الغلاف يستخرج توكن
  Expo Push ويحقنه لصفحة اللوحة فتسجّله بجلستها (`POST /v1/merchant/devices/push-token`)،
  والخادم يرسل الإشعار عند كل طلب يصل الفرع (حدث `merchant.order_received`).
  يتطلب `eas init` (لكتابة projectId) قبل البناء — التفاصيل في `HUMAN-ACTIONS.md` بند B8.

> هذان التطبيقان **خارج مساحة عمل pnpm** عمداً (لا يُبنيان مع الخدمات الخلفية ولا يمسّان خط الإنتاج/CI).
> لكلٍّ منهما `node_modules` و`package-lock.json` مستقلّان.

## أولاً: خطوة لمرة واحدة (منك أنت — لأني لا أستطيع إدخال بيانات دخولك)

البناء يتم على **سحابة Expo مجاناً** (جهازك لا يحتاج Android SDK). سجّل الدخول مرة واحدة:

```powershell
npm install -g eas-cli
eas login
```

- إن لم يكن لديك حساب Expo: أنشئه مجاناً من https://expo.dev/signup ثم `eas login`.
- بعد نجاح الدخول تبقى الجلسة محفوظة على الجهاز، وأستطيع أنا تشغيل بقية الأوامر نيابةً عنك.

## ثانياً: بناء APK للأندرويد (للتجربة قبل الستور)

لكل تطبيق:

```powershell
# تطبيق العميل
cd "mobile-apps/customer"
eas init                 # مرة واحدة: ينشئ مشروع EAS ويكتب projectId
eas build -p android --profile preview

# تطبيق الفرع
cd "../branch"
eas init
eas build -p android --profile preview
```

- ملف `preview` يُنتج **APK** (لا AAB) — مناسب للتثبيت المباشر والتجربة.
- عند الانتهاء يطبع EAS **رابط تحميل** لملف `.apk`. نزّله على جوال/تابلت أندرويد وثبّته
  (فعّل «السماح بتثبيت التطبيقات غير المعروفة» للمتصفح/الملفات).
- اختصار: من داخل مجلد التطبيق `npm run build:apk`.

## ثالثاً: آبل (آيفون / آيباد)

الإعداد جاهز ١٠٠٪. لكن **قوانين Apple تفرض حساب Apple Developer باسمك (99$/سنة)** لتثبيت التطبيق
على جهاز آيفون/آيباد حقيقي أو رفعه للـApp Store — هذا قيد من آبل نفسها:

```powershell
cd "mobile-apps/customer"
eas build -p ios --profile preview   # يسألك عن حساب Apple للتوقيع
```

عند الرفع للـApp Store لاحقاً: التطبيق يعرض تجربة ويب داخل غلاف، وقد تطلب Apple (بند 4.2 «الحد الأدنى للوظائف») مزايا أصلية إضافية:

- **تطبيق العميل**: يملك مزيّة أصلية (الموقع الجغرافي لتدفّق «وصلت») تقوّي القبول.
- **تطبيق الفرع**: يملك الآن مزيّتين أصليتين (إشعارات الطلب الجديد عبر expo-notifications + إبقاء الشاشة صاحية) —
  وهو مع ذلك تطبيق تشغيل داخلي للفريق، والمسار الأنسب توزيعه عبر
  **Apple Business Manager (تطبيق مخصّص/غير مُدرَج)** حيث يُخفَّف بند 4.2، لا عبر المتجر العام.

## البناء التلقائي بعد تسجيل دخولك

بعد أن تشغّل `eas login`، يكفي أن تقول لي «سجّلت الدخول» وأتولّى أنا:
`eas init` + `eas build` للتطبيقين، وأتابع البناء وأعطيك روابط APK الجاهزة.

## تحديث المحتوى
التطبيقان يعرضان الموقع الحيّ مباشرةً — أي تحديث تنشره على الويب يظهر فوراً داخل التطبيق **دون إعادة بناء**.
تحتاج إعادة بناء APK فقط عند تغيير الأيقونة/الاسم/الأذونات/عنوان الموقع.
